## 基于地图可视化的 AI 个性化旅游路线生成系统 

## 需求规格说明书 

1. 引言 

- 1.1 项目背景 

随着国内旅游市场消费升级，自由行、定制化出游已经成为大众出游的主流方 式。大量用户对符合自身偏好、时间预算的个性化旅游路线有着强烈需求，但 目前主流旅游平台提供的多为标准化固定路线，无法满足用户差异化需求，用 户自行规划路线需要耗费大量时间搜集整合信息，时间成本极高。为解决这一 痛点，拟开发本系统，通过 AI 算法结合地图可视化，为用户快速生成个性化定 制旅游路线，降低用户规划成本。 

## 1.2 文档范围 

本文档定义了基于地图可视化的 AI 个性化旅游路线生成系统的全部功能需求、 非功能需求、外部接口需求以及需求验证标准，是后续系统设计、开发、测试 及项目验收的核心依据。 

- 1.3 定义与缩写 

术语/缩写 定义说明 

AI 人工智能，本系统指基于大语言模型的路线智能生成算法 

OTA 在线旅游平台，指提供旅游产品预订与信息服务的第三方平台 

API 应用程序编程接口，本系统指第三方地图、大模型服务提供的开发接口 

GIS 地理信息系统，本系统指实现地图可视化与路径计算的相关功能 

## 2. 总体描述 

## 2.1 产品视角 

本系统为面向 C 端用户的 Web 端应用，不依赖客户端安装，用户通过浏览器即 

可访问使用。系统整体分为三个层级： 

前端展示层：负责用户交互输入、路线地图可视化展示、用户信息管理 业务逻辑层：负责 AI 路线生成逻辑、数据筛选整合、路径计算 

数据层：负责结构化景点数据存储、用户偏好数据存储、第三方 API 对接 

系统依赖第三方开放 API 提供地图服务与大模型能力，不需要从零开发底层地 图与 AI 模型，架构轻量，适配学生项目开发规模。 

## 2.2 用户角色 

本系统包含两类用户角色： 

普通用户：系统的服务对象，可完成偏好输入、路线生成、路线查看、路线保 存操作，无系统管理权限 

管理员用户：负责系统数据维护、景点信息更新、处理用户报错反馈，拥有后 台管理权限 

## 2.3 运行环境 

客户端：支持主流现代浏览器（ Chrome≥90 、 Edge≥90 、 Firefox≥88 、 

Safari≥14），移动端浏览器可适配基础功能 

服务器端： Linux 内核轻量云服务器，支持 Node.js 与 Java 运行环境，内存 

≥2G，带宽≥1M 

## 3. 具体需求 

## 3.1 功能需求 

## 3.1.1 用户端功能 

## 用户偏好输入模块 

支持用户以自然语言输入出行需求，包含出行目的地、出行天数、预算范围、 兴趣偏好、出行人群、特殊约束等信息 提供可视化筛选器作为补充选项，支持用户快速选择出行天数、人均预算、出 

行类型（亲子游/亲子游/休闲游/深度游等）、是否避开热门景点等参数 需求提交前支持用户预览并修改输入内容 

## AI 路线生成模块 

接收用户需求后，自动筛选符合所有约束条件的景点 

结合大语言模型能力，按照时间逻辑对景点进行排序，分配每日行程与推荐停 留时长 

调用地图 API 计算景点间的交通时间，调整行程合理性，生成最终路线方案 

生成失败时，向用户返回清晰的错误提示，引导用户调整需求重新生成 

## 地图可视化展示模块 

在交互式地图上标注路线所有景点的位置，用清晰线路连接每日行程节点 支持用户点击景点标注，查看景点详细信息（开放时间、票价、特色介绍、周 边配套） 

支持地图缩放、平移、拖拽等基础交互操作，支持一键切换不同日期的行程展 示 

同时提供文字版行程列表，方便用户查看行程时间安排 

用户个人中心模块 

支持用户通过手机号验证码注册登录 

支持用户保存已生成的路线，方便后续查看与修改 支持用户保存个人出行偏好，下次生成路线时可快速复用 支持用户对错误的景点信息提交报错反馈 

## 3.1.2 管理端功能 

景点数据管理模块 

支持管理员新增、编辑、删除景点信息，包含景点名称、位置坐标、开放时间、 票价、标签、介绍等字段 

支持按照城市、标签筛选景点数据，支持批量导入导出景点数据 用户反馈处理模块 

支持管理员查看用户提交的景点信息报错反馈，标记已处理/未处理状态 

处理完成后可反馈给前端用户 系统基础设置模块 支持管理员查看系统访问量、路线生成次数等基础统计数据 支持调整系统基础参数（如单日最多推荐景点数量、最大出行天数） 

3.2 非功能需求 

性能需求：单次路线生成响应时间不超过 15 秒，地图加载完成时间不超过 3 秒， 支持 50 人同时在线访问无明显卡顿 

易用性需求：核心操作（输入需求→生成路线→查看路线）不超过 3 步完成， 界面布局清晰，普通用户不需要学习即可上手操作 

兼容性需求：支持 1920×1080 及以上桌面分辨率，适配常见移动端屏幕尺寸 

可维护性需求：系统代码采用分层架构，模块间低耦合，景点数据与业务逻辑 分离，方便后续更新景点数据与功能迭代 

## 3.3 外部接口需求 

地图服务接口：对接高德地图开放平台 JavaScript API v3 版本，实现地图展示、 坐标标注、路径规划、POI 搜索功能 

大语言模型接口：对接通义千问开放 API/GPT-3.5 Turbo API，实现自然语言需 求理解与路线排序功能 

数据接口：前端通过 HTTP RESTful API 与后端进行数据交互，接口返回格式为 

JSON，错误码定义清晰
